📚 [BoooknTalk] Project Master Context (AI 리마인드용)
1. Project Identity (프로젝트 정체성)
서비스명: BoooknTalk (절대 한글 표기 '북앤톡'을 혼용하지 않으며, 반드시 영문 BoooknTalk으로 표기)

서비스 목적: 단순한 독서 관리를 넘어, 책에서 얻은 '사색'과 '영감'을 기록하고 나누는 유료 상용화 기반의 프리미엄 독서 기록 서비스

핵심 철학: 직관적이고 세련된 UI/UX, 사용자 데이터 단위별 세밀한 권한 제어, 파편화된 정보를 논리적으로 그룹화하여 스크롤/높이 부담을 최소화하는 디자인

2. Development Methodology (개발 7단계 프로세스)
나의 작업 방식은 아래의 순서를 엄격하게 따른다.

전체 서비스의 화면 정리

화면의 개발 우선순위 산정

각 화면의 기능 정의

화면의 구성 컨텐츠 정의

필요한 DB 테이블 및 스키마 정의

화면 개발(프론트엔드) 및 백엔드(API) 연동 개발

테스트 및 QA (전체 플로우 검증)

3. Tech Stack & Environment (기술 스택 및 환경)
Frontend: Next.js (React), Tailwind CSS, shadcn/ui (Radix UI 기반)

Backend: Python, FastAPI, SQLAlchemy (ORM)

Database: PostgreSQL

Environment: 로컬 직접 구동 (uvicorn 및 npm run dev) / VS Code / GitHub

Open API: 국립중앙도서관(NLK) API (핵심 서지정보), 네이버 도서 검색 API (표지/소개글 보완)

4. Core Policy: Privacy & Sharing (콘텐츠 공개/비공개 정책)
데이터베이스 설계 및 API 응답의 핵심 기준

고정 공개 (Always Public): 별점(Rating), 태그(Tags)

고정 비공개 (Always Private): 독서 상태(Wish, Reading, Completed), 독서 기간, 현재 진행률 등

사용자 선택 1 (한줄평): Record(user_library) 테이블의 is_short_review_public (Boolean) 컬럼으로 독립 제어

사용자 선택 2 (상세 기록): Memo(posts) 테이블의 is_public (Boolean) 컬럼으로 개별 메모마다 독립 제어

미결정 사항: 기획 및 개발 진행 상황에 맞춰 유연하게 결정 (Agile 대응)

5. UI/UX & Terminology Standards (디자인 및 용어 규격)
간격 규격: 섹션 간 기본 여백은 --spacing-1cm (fallback: 40px~48px)를 준수하여 시각적 일관성 유지.

용어 통일: 짧은 감상평은 무조건 **한줄평**으로 통일 (리뷰, 심층 리뷰 등의 용어 혼용 금지).

공간 최적화: 성격이 비슷한 데이터는 논리적 그룹(Group A, B, C, D)으로 묶고, Inline Text 및 가로 배치(Grid/Flex)를 적극 활용하여 수직 높이 최소화.

직관성: 공개/비공개 제어 등 사용자의 상태 변경 액션은 텍스트가 아닌 '토글 스위치', '명확한 아이콘(자물쇠/지구본)' 등으로 즉각적인 인지가 가능하게 구현.

6. Current Status & Next Steps (현재 상태 및 향후 방향성)
완료된 주요 작업:

도서 검색(2 Tier 아키텍처) 및 서재 등록 플로우 구축

내 서재 목록(Library) 최신순 정렬 및 도서 카드 UI (Hover 메뉴 투명도/색상 최적화)

도서 상세/수정 폼(BookDetailForm) UI 높이 최적화 (날짜/진행률 Grid 배치, 직관적 스위치 토글 적용)

데이터베이스에 공개/비공개 컬럼(is_short_review_public, is_public) 반영 및 API 통신 매핑 완료

도서 상세 페이지 내 타 회원들의 '공개된 한줄평' 모아보기 탭 구현

현재 개발 단계: 7단계. 테스트 (QA 점검 및 전체 플로우 검증) 진행 중

향후 방향성:

개별 도서의 데이터 CRUD(생성, 읽기, 수정, 삭제) 무결성 최종 테스트

'기억의 지층(Memo)' 공개된 메모들의 커뮤니티 공유 뷰(View) 구상

상용화 수준(Paid Service)에 걸맞은 결제/구독 모델 DB 설계 준비 (User 테이블의 is_premium 속성 활용 예정)

7. AI Action Guidelines (AI 작동 지침)
추측 금지: 확신이 서지 않거나 기획 의도가 불분명한 경우 절대 임의로 코드를 작성하지 말고 사용자에게 질문할 것.

오류 점검: 코드나 솔루션을 제시한 후, 스스로 논리적 모순이나 오타가 없는지 한 번 더 검증할 것.

코드 분석 우선: 새로운 기능 추가 시, 관련된 기존 소스 파일 구조를 먼저 파악하고 컨벤션에 맞춰 코드를 생성할 것.
